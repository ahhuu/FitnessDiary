## 🛠️ 第一部分：通用视觉表现与样式统一

### 1. 列表项图标大统一（去除杂乱与“山寨感”）
* **现状痛点**：当前“我的”列表内图标混杂了黄色小人 Emoji、彩色爱心 Emoji、线性放大镜和拟物手表，像素风格与表达维度极度不统一，界面廉价感较强。
* **重构方案**：全量剔除 Emoji 和拟物彩图。全部统一替换为 **Material Design 官方的单色线性线框图标**（`@drawable/ic_...`），颜色与 App 的主题色（如 `@color/fitnessdiary_primary`）保持严格一致。
* **预期视觉效果**：列表视觉重量趋于平稳，符合现代极简、商务健康的 MD3 视觉标准。

### 2. 头像卡片信息层级与对齐优化
* **现状痛点**：等级标签（如 `Lv.3 塑形精英`）字体颜色过浅、可读性低，且在原布局下与昵称在垂直轴线上存在错位，显得局促无序。
* **重构方案**：
  - **色彩对比度提升**：调深标签文本的文字颜色，提升阅读对比度。
  - **标签胶囊化（Badge）**：在布局中为该 TextView 增加轻量拟物背景（例如浅色圆角胶囊背景，圆角半径 `4dp`，内衬微弱内阴影）。
  - **横向对齐**：使用 `LinearLayout` 水平排列或 `ConstraintLayout`，将该胶囊标签紧跟在昵称右侧，并设置 `gravity="center_vertical"` 确保在同一水平线上绝对对齐。
* **预期视觉效果**：等级标签清晰突出，首屏卡片重心稳定。

### 3. 数据层致命伤：脂肪指标与多端数据同步
* **现状痛点**：
  1. 首页饮食卡片的“脂肪”进度由于数据为 0 导致右侧灰色背景轨道完全缺失。
  2. 首页数据（如体重、围度）与详情页修改的数据存在不同步的 Bug。
* **重构方案**：
  - **补齐灰色轨道**：即便脂肪摄入量为 0，底层的灰色轨道 `ProgressBar` 也必须强制完整渲染（进度设为 `0`），保证三道轨道的视觉对称性。
  - **统一数据源（SSOT）**：
    > [!IMPORTANT]
    > 首页大卡片和身体数据详情页**禁止各自独立读取/维护本地状态**。必须统一通过底层 `Room` 数据库提供响应式数据。
    > 一旦用户在详情页或弹窗中修改了数据，首页通过 `LiveData` 或 `Flow` 观察者机制自动触发无感刷新。
  - **补齐脂肪存储**：在食物实体 `FoodRecord` 及关联 of 数据库 DAO 中补齐脂肪（Fat）字段的写入与累加计算，确保其与蛋白质、碳水化合物采用完全一致的数据流进行渲染。

### 4. 善用「卡片分组（Grouping）」建立物理边界
* **现状痛点**：当前页面从“阶段健身目标”到下方的资产、设备、设置，全部采用了一模一样的通栏长条，且没有用清晰的物理容器进行逻辑归类。用户的视线缺乏落脚点，容易产生视觉疲劳。
* **重构方案**：按照功能大类，使用 `MaterialCardView` 或者是定义好的圆角卡片背景，将属于同一类别的行（Row）包裹在一个个独立卡片组中，卡片与卡片之间留出 `16dp` 的垂直间距。
* **预期视觉效果**：页面板块结构一目了然，逻辑层次清晰，彻底告别单调的长列表平铺。

### 5. 建立「严格的字阶与颜色梯队（Typography）」
* **现状痛点**：文字排版杂乱的根源是字号和字色过于接近，没有拉开明确的信息阶梯，导致重要和次要信息混杂在一起。
* **重构方案**：在项目 XML Style（`styles.xml`）和布局中规范字色不要超过 3 种，字号不要超过 4 种，形成严格的字阶梯队：
  - **大标题（昵称）**：`20sp`，加粗（`textStyle="bold"`），主色字（`@color/text_primary`）。
  - **卡片组标题**（如“我的内容资产”）：`14sp`，加粗，中灰色（`@color/text_secondary`），专门用于引领区域物理分隔。
  - **列表项左侧文本**（如“我的常用食谱”）：`16sp`，常规体（`normal`），深灰色（`@color/text_primary`）。
  - **列表项右侧提示**（如“待开发”）：`14sp`，常规体，浅灰色（`@color/text_hint`），使其淡出视觉重心，减少视觉噪音。
* **预期视觉效果**：信息具备强烈的视觉层次感，用户能在极短时间内扫视到核心内容。

### 6. 用「无色线框（Outlined）」或「微弱色差」代替强分割线
* **现状痛点**：目前列表里部分列表项下方隐约能看到一条强对比度的细分割线。如果每个列表项都加一条生硬的横线，页面就会被切得极其零碎。
* **重构方案**：
  - **对比度微调**：每个独立分组卡片背景色使用纯白（`#FFFFFF`），而 App 整体大背景采用微暗的浅灰底色（`#F7F7F7`），以此微弱色差自然衬托出物理卡片边界。
  - **隐形分割**：卡片内部的列表项之间，直接靠 padding 产生的留白来区分，或者使用透明度极低的浅灰分割线（`#1A000000`）。高级感来自于“隐形分割”，而非强行画横死线。
* **预期视觉效果**：界面保持极高通透感，利用留白形成隐形边界。

### 7. 统一右侧交互指示器（Chevron Icon）
* **现状痛点**：有的行右侧写着“配置”，有的写着“待开发”，有的是个箭头 `>`，形式不统一且色彩混乱。
* **重构方案**：
  - **统一小箭头**：凡是点击可以跳转或弹窗的行，右侧一律统一放置一个浅灰色向右小箭头图标（`@drawable/ic_chevron_right`）。
  - **未开启置灰**：对于属于“待开发”状态的行，将其整行的文字及图标透明度设为 `alpha = 0.5f`（置灰表现），同时移除右侧冗余文案，保持干净的视觉呈现。
* **预期视觉效果**：操作意图明确，未开启功能不再干扰主要视觉焦点，页面元素和谐大一统。

## 📐 第二部分：我的页面身体数据中心——「数据」标签页重构

### 身体数据页布局演进图：
```mermaid
graph TD
    subgraph 重构前 "重构前 (组件割裂 / 布局杂乱)"
        A[顶部: 孤立的核心数据卡片]
        B[中部: 单薄的线框火柴人]
        C[两侧: 错落无序的围度输入框]
        A -.- B
        C -.- B
    end
    subgraph 重构后 "重构后 (身体资产看板)"
        D[包裹在同一个流式 ScrollView 内]
        subgraph Canvas "看板整体 Container"
            E[三合一核心大卡片: 体重/目标/体脂率]
            F[中部: 半透明人体剪影 Silhouette]
            G[双侧虚线引线精准指向对应部位]
            H[端正排列的数据组件 标签/数值/单位]
            E --> F
            F --> G
            G --> H
        end
    end
```

### 1. 视觉重心：升级人体剪影
* **现状痛点**：中部的火柴人图形过于抽象、线条纤细苍白，视觉分量太轻，撑不起页面中心，导致界面整体呈现出“工程毛坯感”。
* **重构方案**：移除极简的单薄线条火柴人，更换为一个**具备一定体量感的半透明或纯色极简人体身形剪影**（Silhouette）矢量图作为页面的核心视觉锚点。
* **预期视觉效果**：页面中心视觉力量增强，人体意象更为专业、具象。

### 2. 双侧引线对齐排版
* **现状痛点**：两侧的“脖围、胸围、腰围”等数据框分布杂乱，缺乏视觉栅格对齐，像贴在人体图上的零散补丁。
* **重构方案**：
  - **引线关联**：以中间的人体剪影为中心，从脖子、胸部、腰部、大腿等核心测算部位向两侧拉出精致的**浅灰色水平虚线（引线）**。
  - **数据对齐**：虚线外侧整齐对齐一列标准化的数据组件（采用上下结构：上方为灰色微字标签，下方为黑色大字数据加单位，如：`胸围 / 88 cm`），彻底消除散落感。
* **预期视觉效果**：引导视线由人体部位自然过渡到数据数值，视觉动线清晰规整。

### 3. 结构整合：消除组件剥离感
* **现状痛点**：顶部的三个核心大卡片（当前体重/目标体重/体脂率）与下方的围度人体图层完全割裂，不符合“看板”的整体感。
* **重构方案**：通过统一的卡片背景色或一致的边框，将顶部大卡片与下方人体围度区域包裹在**同一个可滑动的卡片容器（ScrollView 内衬 CardView）内**，形成一个有机的“个人健康资产面板”。
* **预期视觉效果**：界面模块性更强，滑动时呈现整体推移的顺滑感。

## 📊 第三部分：身体数据中心——「图表」标签页重构

### 图表页布局重构对比：
```mermaid
graph LR
    subgraph 旧版 "旧版布局 (憋屈/反向)"
        A1[左侧纵向文字边栏] --> A2[右侧折线图 - X轴时间倒流 / Y轴贴边]
    end
    subgraph 新版 "新版布局 (全屏大气)"
        B1[顶部: 横向滚动微型标签栏 TabRow]
        B2[下方: 舒展大折线图 - 贝塞尔平滑曲线]
        B3[底部: 浅色渐变投影区 + 上下10%动态边距]
        B1 --> B2
        B2 --> B3
    end
end
```

### 1. 逻辑纠错：彻底颠倒的 X 轴时间线
* **现状痛点**：
  > [!CAUTION]
  > 这是一个重大的逻辑 Bug：当前图表的横轴从左到右显示为 `6/30 -> 6/8`，时间线发生倒流，导致折线走势完全反向。
* **重构方案**：修正底层 Room 查询或数据源处理处的排序规则。**SQL 查询或 List 排序时必须使用升序（ASC）**，使得时间轴从左（过去旧日期）向右（现在新日期）严格递增（即 `6/8 -> 6/30`）。
* **预期视觉效果**：折线图的起伏方向符合人类“从过去走向未来”的正常时间认知。

### 2. 交互与视觉：图表增加平滑度与呼吸感
* **现状痛点**：折线图折角极其生硬，且 Y 轴数据点容易顶格贴边（例如最高点几乎被图表最顶部的边框切断），整体画面局促粗糙。
* **重构方案**：
  - **贝塞尔曲线平滑**：若为自定义绘制 Canvas，在构建 Path 时使用 `cubicTo()` 代替传统的 `lineTo()`，使折线过度平滑。若使用第三方图表库（如 MPAndroidChart），开启 `mode = LineDataSet.Mode.CUBIC_BEZIER`。
  - **动态 Padding 机制**：在计算 Y 轴的最大值和最小值时，自动在此区间上下留出 **10% 的动态缓冲区（Padding）**，确保最高和最低的数据点永不顶格或贴边。
  - **面积渐变填充（Gradient Fill Area）**：在折线下方至 X 轴的闭合区域内，填充一层**从半透明主题色自然淡化至全透明的渐变阴影**。
* **预期视觉效果**：图表曲线饱满平滑，数据点具有呼吸感，呈现媲美专业运动软件的高级视觉质感。

### 3. 空间释放：精简左侧边栏
* **现状痛点**：左侧一长串纵向排列的维度切换选项（体重、体脂率、腰围...）占据了横向近 1/3 的空间，导致右侧图表显示区域极为憋屈局促，曲线细节难以看清。
* **重构方案**：**彻底砍掉左侧的纵向侧边栏**。将其重构为页面顶部的**横向滚动微型标签栏（Horizontal TabLayout 或是 HorizontalScrollView 包裹的 ChipGroup）**。用户切换指标时直接横滑点选，将下方 100% 的横向宽度完全释放给图表。
* **预期视觉效果**：图表在横向空间上完全伸展开来，折线波动细节一目了然，横屏或大屏适配度极大提升。
